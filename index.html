Pergunta 1

Quando falamos em versionamentos de códigos também estamos incluindo os commits. Os Commits adicionam as alterações mais recentes no código fonte direto no repositório tornando as alterações parte da revisão do repositório principal. Para fazer commits e necessário aprender alguns comandos para usar direto no Git.


Para commitar no git so é preciso executar um simples comando: git commit –m “Mensagem a ser comitada”


O jeito Certo de fazer commits é fazendo commits semânticos. Os commits semânticos é uma forma que se pode fazer a padronização dos próprios commits utilizando regras simples e claras. Assim quando as pessoas que forem analisar os seus commits poderão entender melhor as alterações feitas por ele.

Pergunta 4

A tag é em geral usado para capturar um ponto no histórico que é usado para uma versão marcada, como por exemplo: "v2.0.2", sendo esse marcador sem ramificações e não tendo histórico de commits. No Git é suportado dois tipos diferentes de marcações, as anotadas e as leves, as marcações de tipo leve e as anotadas diferem na quantidade de metadados acompanhantes que elas armazenam, uma prática recomendada é considerar as marcações anotadas como públicas e as marcações leves como privadas, as marcações anotadas elas armazenam metadados extras como e-mail e data, as marcações leves são em essência marcadores em um commit, são somente um nome e um ponteiro para um commit, úteis para criar links rápidos para commits relevantes.

Pergunta 5

O comando branch um comando que permite criar, listar, renomear e excluir ramificações, é útil em situações nas quais você deseja adicionar um novo recurso ou corrigir um erro, gerando uma nova ramificação garantido que o código instável não seja mesclado nos arquivos de um projeto principal, apenas depois ele pode concluir a atualização dos códigos da ramificação, você pode mesclar a ramificação com a principal, geralmente chamada de master, além disso a branch não permite alternar entre as ramificações ou reunir um histórico bifurcado de novo, diante disso o branch é muito integrado com os comandos git checkout e git merge.

Pergunta 6

Conflitos podem acontecer tanto ao mesclar branches, quanto ao mesclar commits dentro da mesma branch, um termo simples disso é haver um conflito de merge no git ocorre quando dois desenvolvedores alteram o mesmo trecho de código e a única maneira de resolver este conflito é através de uma intervenção manual, alterando o código em questão se submetendo um novo commit. Um meio de resolver conflitos precisa-se encontrar os arquivos onde ocorreram os conflitos, como por exemplo: CONFLICT (contente): Merge conflict in index.html, no final da linha o arquivo onde teve conflito aparecera, para resolver o conflito, é necessário alterar o arquivo para que contenha apenas uma versão, deletando o código incorreto e removendo as marcações de conflito.
